# 接口字段与数据库映射核对清单

## 说明

- 核对范围：`docs/api` 全部接口文档、`src/init.sql`
- 核对目标：请求/响应字段是否可在数据库中一一落地（直接字段或可推导字段）
- 结论分级：
  - `已对应`：数据库已有直接字段或明确可推导
  - `需补充`：数据库缺字段/缺表，无法稳定落地
  - `需统一`：接口语义与数据库语义冲突，建议统一口径

---

## 1. 租户管理（`tenant`）

### 已对应

- `tenantCode` <-> `tenant.tenant_code`
- `tenantName` <-> `tenant.tenant_name`
- `logoUrl` <-> `tenant.logo_url`
- `status` <-> `tenant.status`
- `contactName` <-> `tenant.contact_name`
- `contactPhone` <-> `tenant.contact_phone`
- `inviteCode` <-> `tenant.invite_code`
- `apiQuota` <-> `tenant.api_quota`
- `apiUsed` <-> `tenant.api_used`
- `expireTime` <-> `tenant.expire_time`
- `createdAt` / `updatedAt` <-> `tenant.created_at` / `tenant.updated_at`

### 需补充（可推导但非直接字段）

- `shopCount`：需按 `shop.tenant_id` 聚合统计
- `userCount`：需按 `sys_user.tenant_id` 聚合统计

---

## 2. 门店管理（`shop`）

### 已对应

- `shopCode` <-> `shop.shop_code`
- `shopName` <-> `shop.shop_name`
- `province/city/district/address` <-> 同名蛇形字段
- `managerName` / `managerPhone` <-> `shop.manager_name` / `shop.manager_phone`
- `meituanId` / `eleId` <-> `shop.meituan_id` / `shop.ele_id`
- `status` <-> `shop.status`
- `createdAt` / `updatedAt` <-> `shop.created_at` / `shop.updated_at`

### 风险点

- `shopCode` 文档声明自动生成，数据库仅有唯一索引约束；生成规则需在服务层保证。

---

## 3. 用户管理与登录（`sys_user` + `sys_role`）

### 已对应

- `username` / `realName` / `phone` <-> `sys_user.username` / `real_name` / `phone`
- `roleCode` <-> `sys_user.role_code`
- `status` <-> `sys_user.status`
- `lastLoginTime` <-> `sys_user.last_login_time`
- `createdAt` / `updatedAt` <-> `sys_user.created_at` / `sys_user.updated_at`
- `roleName`：可通过 `sys_role.role_name` 关联得到

### 可推导字段

- `shopIds` <-> `sys_user.shop_ids`（JSON）
- `shopNames`：需按 `shopIds` 关联 `shop.shop_name` 组装
- `tenantName`：需通过 `tenant_id` 关联 `tenant.tenant_name`

### 需统一

- 登录文档请求参数写“密码 MD5 传输”，数据库注释为“BCrypt 存储”，两者不冲突但建议在文档明确“传输层加密/摘要”和“存储层加密”是两件事，避免误解。
- `super_admin` 用户 `tenant_id = NULL`，涉及租户维度查询时需统一空值处理规则（平台态）。

---

## 4. 评价管理（`review` + `review_tag` + `reply_template`）

### 已对应

- `platform` / `platformOrderId` / `platformReviewId` <-> `review.platform` / `platform_order_id` / `platform_review_id`
- `shopId` / `tenant_id`（内部）<-> `review.shop_id` / `review.tenant_id`
- `starRating` / `content` / `orderTime` <-> `review.star_rating` / `content` / `order_time`
- `replyContent` / `replyTime` <-> `review.reply_content` / `reply_time`
- `tags` <-> `review.tags`（JSON）
- `createdAt` / `updatedAt` <-> `review.created_at` / `review.updated_at`
- 标签接口：`review_tag` 表可支撑（`name/type/tenant_id`）
- 模板回复：`reply_template` 表可支撑

### 可推导字段

- `replyStatus`：可由 `reply_content` 是否为空推导（0/1）
- `sentimentType`：可由 `star_rating` 区间推导（1-2差评、3中评、4-5好评）
- `shopName`：需关联 `shop.shop_name`

---

## 5. 数据看板（`shop_metric_daily`）

### 已对应

- `date` <-> `shop_metric_daily.stat_date`
- `totalOrders` / `totalAmount` / `totalReviews` / `negativeReviews` / `avgRating`
  <-> `total_order_count` / `total_order_amount` / `total_review_count` / `total_negative_count` / `total_avg_rating`
- `meituan.*` <-> `meituan_*` 字段
- `ele.*` <-> `ele_*` 字段

### 可推导字段

- 同比环比、趋势百分比、排行、时间序列为聚合/计算结果，数据库字段足够支撑。

---

## 6. 工单管理（`ticket` + `ticket_timeline`）

### 当前决策

- 已确定仅保留 `ticket` / `ticket_timeline` 作为工单主模型。
- `work_order` / `work_order_type` / `work_order_log` 已从 `init.sql` 移除。
- 工单接口路径已统一为 `/api/tickets/*`。

### 已对应（按 `ticket*`）

- `id` <-> `ticket.id`
- `typeId` <-> `ticket.type_id`（关联 `ticket_type.id`）
- `status` <-> `ticket.status`（1-待处理，2-处理中，3-待核销，4-已归档，5-已忽略）
- `shopId` / `tenantId` <-> `ticket.shop_id` / `ticket.tenant_id`
- `title` / `description` <-> `ticket.title` / `ticket.description`
- `creatorId` / `assigneeId` <-> `ticket.creator_id` / `ticket.assignee_id`
- `priority` / `images` <-> `ticket.priority` / `ticket.images`
- `reviewId` <-> `ticket.review_id`
- `deadline` / `createdAt` / `updatedAt` <-> `ticket.deadline` / `ticket.created_at` / `ticket.updated_at`
- 处理日志 <-> `ticket_timeline`（`action/operator_id/remark/created_at`）
- 类型管理接口 <-> `ticket_type`（系统预置 + 租户自定义）

### 待继续收敛

- `statusName`、`typeName`、`shopName`、`creatorName`、`assigneeName` 为展示型字段，需通过关联查询或字典映射生成，不是单表直出字段。
- 若后续需要记录“指派历史”，建议在 `ticket_timeline.remark` 之外增加结构化字段（如 `from_assignee_id`、`to_assignee_id`）。

---

## 7. 登录与邀请码（高优先级缺口）

### 需补充（缺表）

- 文档中的邀请码管理接口（生成/列表/作废）含字段：
  - `id`
  - `inviteCode`
  - `shopId`
  - `shopName`
  - `expireTime`
  - `status`（unused/used/expired）
  - `createdAt`
  - `usedTime`
  - `usedUserId`
  - `usedUsername`

`init.sql` 当前没有邀请码记录表，`tenant.invite_code` 仅能表示单个邀请码，无法支撑“多条记录 + 生命周期 + 作废/已使用追踪”。

建议新增表（示例）：

- `invite_code_record`：
  - `id, tenant_id, shop_id, invite_code, status, expire_time, used_time, used_user_id, created_at, updated_at`
  - 唯一索引：`invite_code`
  - 普通索引：`tenant_id, shop_id, status, expire_time`

---

## 8. 总体结论

- **大部分核心业务字段已可落库**（租户/门店/用户/评价/看板）。
- **两个关键阻塞点**需要先定稿：
  1. 工单采用哪套模型（`work_order*` 还是 `ticket*`）；
  2. 邀请码管理需补“记录表”，当前无法完整落地文档接口。
- 建议先完成这两项后，再进入接口开发，可避免后期大规模返工。
